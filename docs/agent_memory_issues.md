# Agent 记忆机制 — 开发 Issue 拆解

> 基于《Agent 记忆机制设计深度研究报告》拆解，按里程碑分阶段组织。
> 建议开发顺序：M1 → M2 → M3 → M4，各里程碑内部 Issue 可并行。

---

## Milestone 1：基础记忆层（核心骨架）

目标：搭建分层记忆架构的骨架，跑通"写入→存储→注入上下文"的最小闭环。

---

### Issue #1：会话上下文滑动窗口管理

**背景**
研究报告第五层（当前会话滑动窗口）指出，ChatGPT 基于 Token 数量管理当前对话上下文，超出上限时丢弃旧消息但保留上层记忆和摘要。这是记忆系统最底层的基础能力。

**目标**
实现基于 Token 计数的滑动窗口，对当前会话消息进行截断管理，确保每轮对话注入的上下文不超过预设 Token 上限。

**具体任务**
- 实现 Token 计数器（支持主流 tokenizer，如 tiktoken / Anthropic tokenizer）
- 实现滑动窗口截断策略：先丢弃最早的用户/助手消息对，保留系统提示和记忆注入区域不被截断
- 支持配置最大 Token 数（默认建议 8K-16K，可按模型调整）
- 截断时输出日志，记录被丢弃的消息数量和 Token 数

**验收标准**
- 单轮对话 Token 数不超过配置上限
- 系统提示 + 记忆区域始终保留，不被截断
- 截断行为对用户透明（不影响对话流畅性）

**依赖**：无（可最先启动）
**预估工作量**：2-3 人天

---

### Issue #2：用户长期事实结构化存储（User Profile Store）

**背景**
ChatGPT 逆向分析的第三层"用户记忆"是结构化的长期事实档案（约33条，包含姓名、职业、目标、偏好等）。报告核心结论之一是：关键事实应该用结构化存储精确读写，不需要向量检索。

**目标**
设计并实现用户长期事实的结构化存储模块，支持 CRUD 操作和精确查询。

**具体任务**
- 设计 Memory Fact 数据模型：
  ```
  {
    fact_id: str,
    user_id: str,
    category: str,         # 如 "personal", "preference", "goal", "skill"
    key: str,              # 如 "name", "role", "preferred_language"
    value: str,
    source: str,           # "explicit"(用户主动告知) / "inferred"(模型推断)
    confidence: float,     # 0-1，推断事实的置信度
    created_at: datetime,
    updated_at: datetime,
    last_accessed_at: datetime
  }
  ```
- 实现存储后端（第一阶段用 Redis Hash 或 SQLite，预留接口方便后续切换）
- 实现 CRUD API：create_fact / get_facts / update_fact / delete_fact
- 实现按 user_id + category 精确查询
- 设定容量上限（建议初始 200 条 / 用户），超限时按 last_accessed_at 清理最久未使用的低置信度条目

**验收标准**
- 单用户可存储最多 200 条事实，CRUD 操作延迟 < 10ms
- 按 category 查询返回结果正确
- 超容量时自动清理策略生效，不丢失高置信度/近期使用的事实

**依赖**：无
**预估工作量**：3-4 人天

---

### Issue #3：会话元数据采集与注入

**背景**
ChatGPT 的第二层"会话元数据"在每次会话开始时注入设备类型、时区、使用习惯等信息，仅注入一次不长期存储，用于实时调整回复风格。

**目标**
实现会话元数据的采集和系统提示注入机制。

**具体任务**
- 定义元数据字段集合（根据业务需求裁剪）：
  - 基础信息：设备类型、时区、语言偏好
  - 会话上下文：当前时间、入口来源
  - 使用统计（可选）：近 7/30 天活跃天数、平均消息长度
- 实现元数据采集接口（从请求头、客户端上报或数据库读取）
- 实现元数据格式化器：将采集到的元数据序列化为适合注入系统提示的文本格式
- 元数据注入到系统提示的固定区域（在用户记忆之前）

**验收标准**
- 每次新会话正确采集并注入元数据
- 元数据不进入长期存储
- 注入后系统提示总 Token 数增加可控（元数据区域 < 200 tokens）

**依赖**：Issue #1（需要与滑动窗口配合计算 Token 预算）
**预估工作量**：2 人天

### Issue #4：上下文组装引擎（Context Builder）

**背景**
研究报告的核心架构思想是分层注入：系统指令 → 会话元数据 → 用户记忆 → 对话摘要 → 当前会话。需要一个统一的组装引擎来编排各层内容。

**目标**
实现上下文组装引擎，按优先级从各记忆层读取内容，在 Token 预算内组装完整的系统提示。

**具体任务**
- 设计分层 Token 预算分配策略：
  - 系统指令：固定预算（如 500 tokens）
  - 会话元数据：固定预算（如 200 tokens）
  - 用户记忆：弹性预算（如 500-1500 tokens）
  - 对话摘要：弹性预算（如 300-800 tokens）
  - 当前会话：剩余全部预算
- 实现 ContextBuilder 类，定义 `build(session_id, user_id, messages) -> prompt` 接口
- 当总 Token 超限时，按优先级从低到高截断（当前会话最先截断，系统指令最后截断）
- 输出组装日志：各层实际占用 Token 数

**验收标准**
- 组装后的 prompt 总 Token 数严格不超过模型上限
- 各层内容按正确优先级排列
- 即使某层内容为空也不影响组装流程

**依赖**：Issue #1、#2、#3
**预估工作量**：3-4 人天

## Milestone 2：智能记忆管理

目标：让记忆系统从"手动存取"进化为"自动提取+智能更新+主动遗忘"。

---

### Issue #5：LLM 驱动的记忆提取器（Memory Extractor）

**背景**
ChatGPT 的记忆触发有两种方式：用户显式说"记住这个"，或模型检测到符合标准的事实且用户未反对。Mem0 框架进一步用 LLM 判断 ADD/UPDATE/DELETE/NONE 操作。

**目标**
实现从对话内容中自动提取可持久化事实的 Memory Extractor。

**具体任务**
- 设计提取 Prompt：输入为最近 N 轮对话，输出为结构化事实列表（JSON 格式），每个事实包含 category/key/value/confidence
- 实现显式触发检测：识别用户"记住xxx"、"我的xxx是xxx"等显式指令
- 实现隐式提取：每轮对话结束后异步运行 LLM 提取，判断是否有值得记忆的新事实
- 提取后调用 Issue #2 的 CRUD API 写入
- 实现去重逻辑：新提取的事实与已有事实做相似度比对（精确匹配 key 或文本相似度 > 0.9），避免重复写入

**验收标准**
- 显式指令"记住我叫张三"后，User Profile Store 中出现对应记录
- 对话中自然提及"我是后端工程师，主要用 Python"，系统自动提取并存储
- 重复提及同一事实不会产生重复记录，而是更新已有记录的 updated_at

**依赖**：Issue #2
**预估工作量**：4-5 人天

---

### Issue #6：LLM 驱动的记忆冲突解决（Memory Reconciler）

**背景**
Mem0 的核心创新是用 LLM 判断新旧记忆的关系：补充、替代还是忽略。研究报告指出"新值覆盖旧值"是记忆系统的关键挑战。

**目标**
实现记忆冲突检测和 LLM 辅助的解决机制。

**具体任务**
- 设计冲突检测逻辑：当新事实的 (user_id, category, key) 与已有事实匹配时触发
- 设计 Reconcile Prompt：将新事实和旧事实同时输入 LLM，输出决策：
  - `UPDATE`：新事实替代旧事实（如"我搬到上海了"替代"住在北京"）
  - `APPEND`：新事实补充旧事实（如新增一项技能）
  - `IGNORE`：新事实不可靠或不重要，忽略
- 记录每次冲突解决的决策日志（旧值、新值、决策、原因），用于后续审计
- 实现 batch reconcile 接口，一次处理多条冲突

**验收标准**
- 用户先说"我住北京"后说"我搬到上海了"，存储中只保留"上海"
- 用户说"我会 Python"后又说"我也会 Go"，两条都保留
- 所有冲突解决有审计日志可追溯

**依赖**：Issue #2、#5
**预估工作量**：3-4 人天

### Issue #7：近期对话摘要生成器（Conversation Summarizer）

**背景**
ChatGPT 的第四层"近期对话摘要"包含最近约 15 条对话的时间戳、标题和用户消息片段。采用预计算摘要替代 RAG 检索，核心优势是零检索延迟。

**目标**
实现对话结束后的异步摘要生成，并维护最近 N 条对话的摘要池。

**具体任务**
- 设计摘要数据模型：
  ```
  {
    summary_id: str,
    user_id: str,
    session_id: str,
    title: str,           # 对话标题（LLM 生成）
    summary: str,          # 200字以内的摘要
    key_topics: list[str], # 关键主题标签
    user_intent: str,      # 用户核心意图
    created_at: datetime
  }
  ```
- 实现摘要生成 Prompt：输入完整对话历史，输出结构化摘要
- 对话结束（或超过 N 分钟无新消息）时异步触发摘要生成
- 维护最近 15-20 条摘要的滑动窗口，超出时淘汰最旧的
- 存储后端用 Redis List 或数据库表，按 user_id 隔离

**验收标准**
- 每次对话结束后 30 秒内生成摘要
- 摘要池始终保持最近 15-20 条
- 摘要内容准确反映对话核心内容，Token 占用 < 800（整个摘要池）

**依赖**：Issue #1（Token 计数复用）
**预估工作量**：3-4 人天

### Issue #8：记忆遗忘与清理机制（Memory GC）

**背景**
研究报告指出遗忘应该是"可控的设计而非不可控的事故"。需要基于时间、重要性和容量三个维度的遗忘策略。ChatGPT 的两次数据丢失事故是反面教材。

**目标**
实现定时运行的记忆清理服务，按策略自动清理过期和低价值记忆。

**具体任务**
- 实现三维遗忘策略：
  - 时间衰减：超过 N 天（可配，默认 90 天）未访问的记忆标记为 stale
  - 重要性评分：基于 source（显式 > 推断）、confidence、访问频率计算综合分
  - 容量限制：单用户超过上限时，按综合分从低到高清理
- 实现 MemoryGC 定时任务（建议每天凌晨运行一次）
- 清理前生成快照备份（防止误删）
- 提供手动触发接口和 dry-run 模式（预览将被清理的记忆）
- 清理日志记录：删除了哪些记忆、原因、时间

**验收标准**
- 90 天未访问且低置信度的记忆被自动清理
- 高置信度/高频访问的记忆不被误删
- dry-run 模式正确预览清理结果
- 清理前快照可用于恢复

**依赖**：Issue #2
**预估工作量**：3 人天

## Milestone 3：高级检索能力

目标：为历史经验类记忆引入语义检索能力，实现混合检索架构。

---

### Issue #9：向量存储集成（历史经验回忆层）

**背景**
研究报告明确指出向量数据库适合"模糊的、开放的、难以穷举的非结构化内容"，如历史工单、过往方案、失败尝试等。这是分层架构中"历史经验"层的存储后端。

**目标**
集成向量数据库，为历史经验类记忆提供语义检索能力。

**具体任务**
- 选型评估（建议 Qdrant 或 Chroma，本地部署优先）
- 设计 Experience 数据模型：
  ```
  {
    experience_id: str,
    user_id: str,
    content: str,          # 经验内容
    embedding: vector,
    metadata: {
      session_id: str,
      topic: str,
      outcome: str,        # "success" / "failure" / "neutral"
      created_at: datetime
    }
  }
  ```
- 实现 Embedding 生成管道（支持 OpenAI / 本地模型）
- 实现写入接口：对话摘要生成后，提取有价值的经验写入向量库
- 实现检索接口：输入查询文本，返回 top-K 相关经验
- user_id 级别的数据隔离

**验收标准**
- 写入 1000 条经验后，语义检索 top-5 准确率 > 80%
- 检索延迟 < 100ms（本地部署）
- 不同用户数据完全隔离

**依赖**：Issue #7（摘要作为经验的来源之一）
**预估工作量**：4-5 人天

---

### Issue #10：混合检索编排器（Hybrid Retriever）

**背景**
生产级系统趋向混合架构。报告建议级联策略：先结构化存储精确查询，未命中再向量库语义匹配。

**目标**
实现统一的混合检索接口，编排结构化查询和向量检索。

**具体任务**
- 设计 HybridRetriever 接口：`retrieve(user_id, query, context) -> MemoryResult`
- 实现级联检索策略：
  1. 先查 User Profile Store（精确匹配 key/category）
  2. 再查近期对话摘要池（关键词 + 时间范围）
  3. 最后查向量库（语义匹配，top-K）
  4. 合并去重，按综合得分排序
- 实现综合评分：权重 = w1 * 精确匹配分 + w2 * 语义相似度 + w3 * 时间衰减分 + w4 * 重要性分
- 结果裁剪：总结果 Token 数不超过记忆区域预算
- 检索结果 tracing：记录每条结果的来源层和得分

**验收标准**
- 精确事实问题（"我叫什么名字"）优先命中 Profile Store
- 模糊问题（"之前讨论过的方案"）能从向量库召回相关经验
- 综合结果 Token 数在预算内
- 完整 tracing 日志可追溯

**依赖**：Issue #2、#7、#9
**预估工作量**：4-5 人天

## Milestone 4：工程化与团队协作

目标：参考 Claude Code 的 CLAUDE.md 机制，实现团队级规则共享和记忆管理工具。

---

### Issue #11：项目规则文件系统（Rules Engine）

**背景**
Claude Code 的 CLAUDE.md 四层层级结构（组织 → 全局 → 项目 → 模块化规则）支撑了团队协作和规范驱动的工作流。可通过 Git 版本控制共享规则。

**目标**
实现基于文件系统的项目规则加载和优先级合并机制。

**具体任务**
- 设计规则文件层级：
  - 组织级：`/etc/agent/rules.md`（或配置中心下发）
  - 用户级：`~/.agent/rules.md`
  - 项目级：`{project_root}/.agent/rules.md`
  - 模块级：`{project_root}/.agent/rules/*.md`（支持 YAML frontmatter 的 paths 字段做路径匹配）
- 实现规则加载器：按层级依次读取，后者覆盖前者
- 实现路径匹配规则：当操作特定类型文件时，自动加载对应的模块规则
- 规则内容注入到系统提示的"系统指令"区域
- 规则文件变更检测：文件修改后下次会话自动生效

**验收标准**
- 四层规则文件正确加载和合并
- 模块级规则的路径匹配正确（如编辑 .py 文件时加载 Python 规则）
- 规则内容正确注入系统提示
- 规则文件可纳入 Git 版本控制

**依赖**：Issue #4（Context Builder 提供注入位置）
**预估工作量**：3-4 人天

---

### Issue #12：Auto Memory 自动积累机制

**背景**
Claude Code 的 Auto Memory 在工作过程中自动记录项目构建命令、常见错误模式、架构决策等。存储在项目级 MEMORY.md，每次会话开始读取。

**目标**
实现类似 Claude Code 的 Auto Memory：Agent 在工作中自动积累项目级别的学习经验。

**具体任务**
- 设计 Auto Memory 存储格式：Markdown 文件，按主题分区（构建命令、错误解决、架构决策、用户偏好）
- 实现写入触发条件检测：
  - 执行了构建/测试命令并成功 → 记录命令
  - 遇到并解决了错误 → 记录错误模式和解决方案
  - 用户做出架构决策 → 记录决策和理由
- 实现去重和合并逻辑：同一命令/错误不重复记录
- 会话开始时读取 MEMORY.md 前 200 行注入上下文
- 提供 `/memory` 命令接口供用户查看和编辑

**验收标准**
- Agent 解决一个构建错误后，MEMORY.md 自动出现对应记录
- 下次新会话涉及同类问题时，Agent 能引用之前的解决方案
- MEMORY.md 内容可读、可编辑、可 Git 跟踪

**依赖**：Issue #5（复用 Memory Extractor 的提取能力）
**预估工作量**：4-5 人天

---

### Issue #13：记忆管理 API 与调试工具

**背景**
研究报告强调记忆系统需要透明性和可控性。开发和调试阶段需要便捷的记忆查看、修改和诊断工具。

**目标**
提供记忆系统的管理 API 和开发者调试界面。

**具体任务**
- 实现 REST API：
  - `GET /api/memory/{user_id}/facts` — 查看用户所有事实
  - `GET /api/memory/{user_id}/summaries` — 查看对话摘要池
  - `GET /api/memory/{user_id}/experiences?q=xxx` — 语义搜索经验
  - `DELETE /api/memory/{user_id}/facts/{fact_id}` — 删除指定事实
  - `POST /api/memory/{user_id}/gc?dry_run=true` — 手动触发清理
  - `GET /api/memory/{user_id}/stats` — 记忆使用统计（各层条目数、Token 占用）
- 实现上下文诊断接口：
  - `GET /api/debug/context/{session_id}` — 查看当前会话的完整上下文组装结果，包含各层 Token 占用
- 实现简易 Web UI（可选，但建议有）：可视化展示用户记忆全景

**验收标准**
- 所有 API 正常可用，响应时间 < 200ms
- 通过 API 能完整查看和管理用户记忆
- 上下文诊断能准确展示各层的 Token 分配

**依赖**：Issue #2、#4、#7、#9
**预估工作量**：4-5 人天
